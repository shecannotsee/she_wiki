# 软件架构图的 C4 分层方法

## 提取结果

- 软件架构图最容易失效的地方，不是图形工具不够强，而是抽象层级、元素含义和关系语义没有被固定下来
- C4 Model 用 `Software System -> Container -> Component -> Code` 这四层固定抽象，给软件团队提供了一套足够小但足够稳定的共同语言
- 真正高价值的图通常不是“画得最细”，而是按受众和问题选择合适缩放层级；对大多数团队，`System Context + Container` 已经覆盖了主要沟通需求
- 画图纪律和建模纪律同等重要：标题、范围、图例、元素类型、职责描述、技术标注、关系方向和关系标签，缺一项都会显著降低可读性
- 组件图和代码图天然更容易过时，因此应按需生成或自动生成，而不是把所有层级都变成手工维护负担

## 稳定结构

1. 先定义系统边界，再识别外部人和外部系统
2. 再在系统边界内识别运行时容器，而不是直接从代码包结构起手
3. 只有在需要解释容器内部职责时，才继续下钻到组件
4. 只有在关键或复杂组件上，才进一步展示代码级结构
5. 每张图都补齐范围、图例、职责说明、技术信息和关系语义
6. 让不同层级之间形成清晰的缩放关系，而不是彼此重复或互相矛盾

## 可直接复用

- 如果一张图里同时出现外部用户、数据库表、部署节点和业务模块，通常说明抽象层级已经混在一起了
- `Container` 先按运行时边界理解，不要先联想到 Docker、JAR、DLL 或文件夹
- 画微服务时先问：这些服务对当前团队来说是系统内部实现细节，还是已经是独立拥有的软件系统
- 画消息架构时，优先把 queue/topic 当作明确的数据存储型构件，避免让“总线大圆圈”掩盖真实耦合关系
- 对长期文档，优先手工维护上下文图和容器图，把组件图和代码图交给静态分析、IDE 或模型工具
- 审查一张架构图时，不先问“好不好看”，先问：
  - 范围清不清楚
  - 元素类型明不明确
  - 关系有没有方向和语义
  - 技术信息是否足够支撑讨论
