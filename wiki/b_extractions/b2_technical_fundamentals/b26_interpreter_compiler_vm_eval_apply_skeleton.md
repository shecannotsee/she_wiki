# 解释器、编译器与虚拟机的求值骨架

## 提取结果

- 学解释器时，最先该抓住的不是某门语言的表面语法，而是 `eval`、`apply`、environment、表达式分类和控制流分派这套骨架
- 解释器、显式控制求值器、虚拟机和编译器虽然层级不同，但都在处理同一件事：把一种表示翻译或变换成另一种可执行过程
- 编译器和解释器的边界要分清：
  - 解释器更直接参与求值
  - 编译器更核心的职责是生成后续执行所需的代码或中间表示
- 环境不是一个附属细节，而是解释器成立的关键：
  - 变量绑定
  - 过程闭包
  - 形参与实参关联
  - 状态跨步骤延续
  - 都要靠环境链承载

## 稳定结构

1. 把输入表示解析成可分类、可递归处理的结构
2. 用 `eval` 判断表达式类型并分派到对应规则
3. 用 `apply` 把过程应用到实参并进入新的环境
4. 用环境链维护绑定关系、闭包上下文和状态延续
5. 若进入更低层执行，再把高层求值逻辑翻译成寄存器、跳转和指令序列

## 可直接复用

- 看一门语言的解释器实现时，先问这些问题：
  - 表达式被分成了哪些类型
  - `eval` 如何区分并处理这些类型
  - `apply` 如何区分原语过程和组合过程
  - 闭包环境如何保存
  - 变量查找如何沿环境链进行
- 学编译器时，不要把“代码生成”当成唯一核心：
  - 更稳定的理解是“编译是翻译，不是直接求值”
  - 它把高层语义转成更低层的表示，再交给后续执行系统运行
- 看虚拟机时，也不要把它理解成和解释器完全割裂：
  - 它只是把求值过程换成更低层的寄存器、跳转、栈或内存模型去承载
  - 本质仍然是在安排过程如何被执行
- 学编译原理或解释器原理时，一个更稳的入口通常是：
  - 先抓语义上的 `求值` / `应用`
  - 再看环境、闭包和控制流
  - 再进入词法、语法、汇编、代码生成和优化这些分层实现
