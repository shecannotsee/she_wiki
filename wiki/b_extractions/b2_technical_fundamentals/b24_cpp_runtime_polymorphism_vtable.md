# C++ 运行时多态的虚表骨架

## 提取结果

- `vtable` 不是 C++ 语言标准强制规定的实体，而是编译器实现运行时多态的常见方案。
- 普通非虚成员函数调用通常可在编译期静态确定目标地址；真正需要额外运行机制的是“父类接口指向子类实现”的动态分派。
- 从编译结果看，类的数据和操作并不是天然绑在一起的；成员函数本质上仍接近“函数 + 对象指针参数”的调用模型。
- 含虚函数的对象通常会在对象头部增加一个 `vptr`，由它把对象实例连接到一张保存虚调用信息的 `vtable`。
- `vtable` 不只是虚函数地址数组；常见实现里还包含 `offset_to_top`、`RTTI/type_info` 等元信息。
- 派生类重写虚函数时，本质上是在自己的虚表槽位里替换对应函数地址；未重写的槽位继续沿用基类实现。
- 运行时虚调用的稳定骨架是：
  - 从对象取 `vptr`
  - 按虚函数槽位定位目标地址
  - 以当前对象指针为隐式参数完成调用
- 虚表前部的偏移与类型信息不是装饰物，它们服务于多重继承中的指针调整和运行时类型识别。

## 稳定结构

1. 先区分“语言层面的虚函数语义”和“编译器层面的具体实现”。
2. 再区分静态绑定与动态绑定，明确只有后者需要运行期查找。
3. 把对象看成“成员数据 + 一个可选的虚表指针”，而不是把类语法直接投射成机器结构。
4. 再理解 `vptr -> vtable -> 函数槽位` 这条运行路径。
5. 最后再进入 ABI、槽位布局、`RTTI`、多重继承和 `thunk` 这些实现细节。

## 可直接复用

- 看到“虚表是不是语言标准的一部分”时，先回答语义和实现是两层事，不要混成一个问题。
- 学多态时，先问“编译器怎么在运行期知道该调哪个函数”，再背 `virtual` 语法。
- 分析对象大小变化时，先检查类是否引入了虚函数，因为这通常意味着对象里会新增 `vptr`。
- 遇到反汇编或调试里的虚调用，优先按“对象 -> vptr -> vtable -> 槽位 -> 函数地址”这条链理解。
- 讨论多重继承和 `RTTI` 时，先意识到虚表里除了函数地址还存在元信息，不要把它简化成纯函数数组。
