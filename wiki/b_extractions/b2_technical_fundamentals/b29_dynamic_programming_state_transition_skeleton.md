# 动态规划的状态、转移与依赖压缩骨架

## 提取结果

- 动态规划首先不是“背公式”，而是把一个问题压缩成可递推的状态空间。
- 最核心的问题不是“套哪道题型”，而是：
  - 什么历史信息必须保留
  - 什么历史细节可以丢弃
  - 当前答案依赖哪些更小或更早的状态
- 一个更稳的设计顺序是：
  - 先定义状态含义
  - 再写状态转移
  - 再补边界条件
  - 最后确认计算顺序和空间占用
- 设计状态时，要让状态刚好携带后续决策所需的最小信息：
  - 不足时，转移写不出来
  - 过多时，状态空间会膨胀
- 动态规划能成立，通常意味着两件事：
  - 后续求解只依赖当前状态值，而不依赖到达该状态的完整历史细节
  - 大问题的答案可以由更小子问题的答案组合出来
- 这就是常说的：
  - 无后效性
  - 最优子结构
- 但在实际建模时，更有操作性的问法是：
  - 如果我只保留这个状态值，后面还够不够用
  - 如果不够，状态定义就还不完整
- 动态规划与暴力的关键区别，不是“都枚举了很多东西”，而是：
  - 暴力枚举完整构造过程
  - 动态规划只保留足以推出答案的状态值
  - 本质上是在缩小搜索和记忆的对象
- 状态转移可以从两个方向设计：
  - `pull`：当前状态从哪些前驱状态推来
  - `push`：当前状态求出后，可以更新哪些后继状态
- 只要依赖关系清楚，这两种视角本质上是在描述同一张状态依赖图。
- 动态规划不只适用于数组下标问题：
  - `DAG` 最短路可以把点当状态
  - `LIS` 可以把“以某位置结尾”当状态
  - 编辑距离可以把“两个前缀长度”当状态
  - 网格问题可以把“到某个坐标时的答案”当状态
- 因此真正稳定的骨架不是题型名，而是：
  - 状态是什么
  - 依赖是谁
  - 聚合方式是求和、取最小、取最大还是别的组合
- 初始值不是形式补丁，而是递推开始的边界。
- 只要边界漏掉、含义不一致，后续所有状态都可能被带偏。
- 计算顺序必须服从依赖方向：
  - 依赖更小规模，就按规模递增
  - 依赖上方和左方，就按行列推进
  - 依赖 `DAG` 前驱，就按拓扑序推进
- 动态规划的空间优化，本质上不是“神奇技巧”，而是依赖压缩：
  - 如果当前状态只依赖前一行或前一层
  - 那么更老的历史层可以丢掉
  - 二维就可能压成一维
- 压缩空间前，要先画清依赖关系：
  - 当前格依赖上和左时，一维覆盖通常可行
  - 当前格还依赖左上旧值时，需要额外临时变量保存被覆盖前的数据
- 因此，二维压一维的真正前提不是“题目看起来像网格”，而是：
  - 依赖范围局部
  - 覆盖顺序不破坏仍要使用的旧值

## 核心结构

1. 先把原问题改写成一个状态定义：`dp[x]` 或 `dp[i][j]` 到底表示什么。
2. 判断这个状态是否已经携带了后续决策所需的最小充分信息。
3. 找出当前状态依赖哪些更小或更早的状态。
4. 写出状态转移方程，并明确聚合方式是 `sum / min / max / count / boolean` 中哪一种。
5. 找到不能继续递推的边界，给出初始值。
6. 选择与依赖方向一致的计算顺序。
7. 再检查是否只需要保留局部历史，从而做空间压缩。

## 可直接复用

- 做动态规划题时，先写四句草稿：
  - `dp[...]` 表示什么
  - 它从哪些状态来
  - 边界状态是什么
  - 按什么顺序填表
- 如果转移写不出来，先怀疑状态定义不对，而不是急着背题型。
- 如果一个状态记录了大量构造过程细节，先怀疑它装了太多不必要的信息。
- 判断能不能压空间，先问：
  - 当前状态最多依赖几层历史
  - 更新时会不会覆盖后面还要用的旧值
  - 是否需要一个额外变量保存对角或上一轮旧值
- 看二维 `dp` 时，可以先画一个小矩阵，在图上标出：
  - 上方依赖
  - 左方依赖
  - 左上依赖
  - 然后再决定一维数组的遍历方向
- 判断一个问题是不是动态规划，优先问这些问题：
  - 它是不是在重复解决相似子问题
  - 子问题答案能否组合成大问题答案
  - 如果只保留状态值，未来是否仍然可算
- 学动态规划时，比起背更多题，更稳的训练方式是：
  - 反复练习状态定义
  - 反复检查依赖图
  - 反复验证边界和更新顺序
