# AST 作为源码结构抽象层的职责

## 提取结果

- `AST` 的核心价值，不是“把代码画成树”，而是把源码从表面文本提升成一种可统一处理的结构化表示
- token 流解决“源码被切成哪些词法单元”，而 `AST` 解决“这些单元如何形成具有语义的层级结构”
- `AST` 之所以是“抽象”语法树，是因为它会主动丢弃许多对后续处理不重要的细节：
  - 空白
  - 分隔符
  - 注释
  - 某些只服务于表面语法的标记
- 一旦程序被表示成 AST，后续的分析、变换和代码生成就可以围绕“节点类型 + 层级关系 + 字段语义”工作，而不必依赖脆弱的字符串匹配

## 核心结构

1. 先把源码做词法分析，得到 token 流。
2. 再做语法分析，把 token 组织成有层级关系的结构。
3. 在这个过程中丢弃不重要的表面语法细节，保留真正有处理价值的结构信息。
4. 用稳定的节点类型表达程序构造，如声明、表达式、标识符、字面量、块、函数等。
5. 让语义分析、代码改写、IR 生成和后续编译阶段都围绕这套结构化表示工作。

## 可直接复用

- 学 AST 时，先分清三层职责：
  - 原始源码文本
  - token 流
  - AST
- 判断某个结构该不该进 AST 时，先问：
  - 它是否影响程序结构理解
  - 它是否影响后续分析或转换
  - 如果只是在表面语法里起分隔作用，通常就不值得原样保留
- 做代码工具时，优先用 AST，而不是直接用正则或字符串替换：
  - 查找某类结构
  - 改写导入
  - 插入节点
  - 删除节点
  - 这些都更适合在 AST 上做
- 看一门语言或一个 parser 的 AST 定义时，重点先看：
  - 节点类型有哪些
  - 每类节点的关键字段是什么
  - 父子关系如何表达
  - 哪些源码细节被保留，哪些被抽象掉了
- 如果再往后学习编译器，可以把 AST 理解为源码进入语义处理前的第一层稳定中间表示：
  - 它还不是底层 IR
  - 但已经不是原始文本
  - 它负责把程序结构变成编译器可操作的数据结构
