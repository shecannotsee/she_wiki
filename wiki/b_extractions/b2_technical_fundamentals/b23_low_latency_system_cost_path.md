# 低延时系统的端到端成本链路

## 提取结果

- 讨论高频交易或类似低延时系统时，首先要优化的不是“并发规模”，而是单次处理路径上的端到端延时。
- 一个看似很短的业务动作，往往跨越：
  - 网卡接收
  - 中断与协议栈处理
  - 内核缓冲区
  - 进程调度
  - 系统调用
  - 用户态 / 内核态切换
  - 数据复制
  - 应用计算
  - 结果发送
- 在低延时场景里，系统调用、上下文切换、数据复制、缓冲区占用时间和调度抖动，都会成为真实成本。
- 语言和框架的价值，不该抽象地讨论“谁更快”，而要看它是否允许开发者更细地控制上述成本。
- 继续往下压延时时，优化常常会从应用层进入网络旁路、内核绕过或更贴近硬件的方案，例如 `DPDK`、`RDMA` 或网卡侧能力。

## 稳定结构

1. 先确认目标指标是吞吐量还是单次延时
2. 如果目标是低延时，就先把完整处理链路按层拆开
3. 标出每一层的固定成本与波动成本
4. 先找系统调用、复制、调度和缓冲区占用这些高频损耗点
5. 再决定是否需要进一步下沉到内核旁路或硬件级优化

## 可直接复用

- 评估“某种语言是否适合低延时系统”时，先问它能否减少不可控运行时开销，而不是先争论语法偏好。
- 看到系统性能问题时，不要直接把原因归到策略逻辑或业务代码，先把网卡到应用再到发送的整条链路画出来。
- 面对需要极低尾延时的系统时，要特别关注：
  - 是否存在不必要的数据复制
  - 是否频繁进入内核
  - 是否依赖不可预测的调度
  - 是否长时间占用共享缓冲区
- 当问题已经进入微秒级竞争时，系统边界会从“程序优化”扩展成“网络、内核、调度、内存、硬件协同优化”。
