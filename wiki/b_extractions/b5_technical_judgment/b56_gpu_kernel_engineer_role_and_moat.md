# GPU Kernel Engineer 岗位的长期价值与护城河

## 提取结果

- 在 AI Infra 里，GPU kernel engineer 持续稀缺，不是因为现成库不存在，而是因为通用库只能覆盖平均情况，生产成本却常被长尾 shape、特殊数据流和新模型结构决定。
- 真正有价值的 kernel 优化，很多时候不是把单个 GEMM 再抠一点，而是围绕 `memory wall`、算子融合和数据流重排减少显存往返。
- 自动编译和代码生成会越来越强，但在关键路径、特殊量化布局、动态索引和跨算子深度融合场景里，人工优化仍经常是最后兜底。
- 这个岗位的长期护城河，不是会不会调某个库，而是能把体系结构、数值稳定性和系统 dataflow 连成一个整体来判断。

## 核心判断顺序

1. 先问当前瓶颈是不是通用库覆盖不到的固定 shape、特殊 layout 或高成本关键路径。
2. 再问问题主要卡在算术吞吐，还是卡在 `HBM / SRAM / register` 之间的数据搬运。
3. 如果瓶颈在搬运和中间结果，优先考虑算子融合、重排和 pipeline，而不是先换库名。
4. 再判断自动生成方案是否已经足够稳定；如果搜索空间、成本模型或硬件细节让结果不可靠，再进入手工 kernel。
5. 最后检查优化是否需要跨算法、框架和硬件协同，而不是把 kernel 当作孤立热点处理。

## 可直接复用

- 当有人说“都有 `cuBLAS/CUTLASS/Triton` 了，为什么还要手写”时，先追问当前 workload 是否落在通用库的长尾区。
- 评估某个 kernel 是否值得手写时，先算集群成本和热点占比，不要只讨论代码优雅度。
- 看到一串前后处理小算子时，先怀疑是不是显存往返过多，而不是默认 ALU 不够快。
- 处理量化推理、`MoE gather/scatter`、特殊 epilogue 或深度融合时，优先预期通用模板和自动生成可能不够。
- 招聘或培养这类工程师时，优先看：
  - 是否真懂内存层级和 warp 调度
  - 是否理解混合精度与数值风险
  - 是否能从 profiling 走到系统级 dataflow 重构
- 做职业布局时，不要把自己锁死在某个库 API 上；更稳的是向下补体系结构，向上补编译器与 workload 理解。
