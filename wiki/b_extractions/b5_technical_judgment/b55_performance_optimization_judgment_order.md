# 性能优化要优先处理 IO、布局与减法空间

## 提取结果

- 真正值得优先做的优化，常常不是代码层的小技巧，而是对数据搬运、存储层级和系统结构的重写。
- 在现代硬件上，很多系统首先受限于 memory wall，而不是算术吞吐。
- 因此判断优化价值时，更稳的顺序通常是：
  - 先看数据有没有在来回搬
  - 再看布局是否顺应缓存和 SIMD
  - 再看是否可以用近似或概率结构换掉昂贵精确解
  - 最后再问能不能直接删掉整段流程
- `Flash Attention` 这类案例说明：
  - 为了减少慢速存储读写
  - 可以接受更多重计算
  - “多算少搬”在现代系统里常常更快
- 这意味着性能优化时，不能默认“减少计算量”就是第一目标。
- `llama.cpp` 这类案例说明：
  - 在资源受限场景下
  - 好优化往往来自按块量化、贴合 SIMD、贴合缓存和目标架构
  - 代码外观可能不优雅，但它顺应了物理约束
- `Data-Oriented Design` 这条线说明：
  - 抽象风格不能压过数据布局
  - 对象散落、指针追逐和随机访问会把缓存命中率拖垮
  - `SoA` 往往比 `AoS` 更适合批量遍历和向量化
- 概率性数据结构说明了另一类高价值优化：
  - 不是把旧方案做得更精
  - 而是直接换一个近似但足够好的问题表述
  - `HyperLogLog`
  - `Bloom Filter`
  - 都是这种思路
- 这些方案的价值在于：
  - 主动放弃不必要的精确性
  - 换来更好的空间、吞吐和系统可扩展性
- 终极优化常常是做减法：
  - 不是把已有环节优化 20%
  - 而是识别某段处理、某个服务跳数、某次清洗或某层架构根本可以删除
- 因此“最好的代码是没有代码”并不是空话，而是一种很严肃的系统判断标准。
- 真正高级的优化，往往具备这些特征：
  - 顺着硬件规律
  - 敢于反直觉权衡
  - 服从数据布局
  - 有勇气砍流程

## 核心判断顺序

1. 先确认瓶颈是不是出在数据搬运、缓存失配和带宽不足，而不是先盯算术指令数量。
2. 再检查数据布局是否顺应访问模式、缓存和 SIMD。
3. 再判断问题是否可以改写成近似估计、概率判定或块级压缩。
4. 再评估是否存在可以直接删除的流程、服务层次或中间产物。
5. 最后才进入热点函数里的局部常数级优化。

## 可直接复用

- 遇到性能问题时，先问：
  - 数据在系统里搬了几次
  - 哪层存储最慢
  - 计算单元是不是一直在等数据
- 看一段高性能代码时，不要先评价“优不优雅”，先问：
  - 它是不是贴着硬件物理规律写的
  - 它是不是在减少缓存 miss 和带宽浪费
- 设计数据结构时，不要只按面向对象的概念分类，先按最常见遍历和更新模式看布局。
- 如果业务只需要近似答案，优先评估：
  - 概率数据结构
  - 分块压缩
  - 近似统计算法
- 当系统链路很长时，先问这一步是否还需要存在，而不是先给它加缓存、并发和线程池。
- 看到“优化得很惊艳”的案例时，优先检查它是不是做到了四件事：
  - 少搬
  - 顺序访问
  - 改问题表述
  - 删环节
