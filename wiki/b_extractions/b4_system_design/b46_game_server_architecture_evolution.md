# 游戏服务端架构的类型分化与演进

## 提取结果

- 游戏服务端架构不应先按“端游/手游”划分，而应先按交互强度、世界模型、同步要求和负载形态划分
- 弱交互游戏通常可以采用简单的 `HTTP + 数据库 + 缓存` 模型；强交互、场景化游戏往往需要长连接、网关、场景服和数据代理；无缝大世界则会进一步引入区域节点、对象层和动态负载均衡
- 从 `MUDOS` 到现代 MMORPG 服务端，演进主线不是“技术栈升级”，而是单体事件循环在内容复杂度、用户规模和世界连续性压力下被逐步拆分
- 架构拆分不是越多越先进；每增加一层服务，性能空间可能上升，但状态机复杂度、研发成本和故障定位成本也会同步上升
- 对战类、战网类和休闲房间类服务端的核心矛盾并不相同：
  - 有的重点在全区匹配和 P2P/NAT 穿透
  - 有的重点在同步一致性与反作弊
  - 有的重点在用户数据并发写控制

## 核心结构

1. 先按交互强度区分：弱交互异步、强交互场景、无缝世界、房间对战、副本实时战斗。
2. 先按世界模型区分：离散关卡、房间/场景切换、连续大地图、全区统一匹配。
3. 先按连接与转发职责拆分：客户端直连业务服，还是通过 `Gate / Session` 统一接入。
4. 再按状态承载位置拆分：场景状态、玩家对象状态、数据库访问、缓存与代理分别落在哪层。
5. 最后再引入动态负载均衡、P2P 中继、对象迁移、反作弊复算等进阶机制。

## 可直接复用

- 评估一个游戏项目的后端骨架时，先问这些问题：
  - 玩家之间是否需要持续实时可见和实时互相影响
  - 世界是离散关卡还是连续地图
  - 是分区分服还是全区统一
  - 每局人数上限有多大
  - 游戏结果更多由服务端裁决还是客户端本地计算
- 如果是弱交互、异步对抗、排行榜和资源结算类游戏：
  - 优先考虑简单的 `HTTP` 服务
  - 优先追求开发速度、可调试性和成本控制
  - 不要过早引入复杂长连接和多层拆分
- 如果是传统 MMORPG 或强交互场景服：
  - 网关负责连接与转发
  - 场景服负责地图/玩法逻辑
  - 数据访问通过缓存与代理降压
  - 这是最常见、最稳的中间形态
- 如果是无缝大世界：
  - 仅按场景切服不够
  - 需要把场景节点管理和玩家对象管理拆开
  - 需要准备区域迁移、边界协作和动态负载均衡机制
- 如果是战网/竞技类：
  - 重点不在数据库，而在匹配、P2P 可达性、同步协议和反作弊
  - 必须提前想清楚 `Host` 模型、投票裁决、复算与作弊记录
- 如果是全区休闲/房间类：
  - 重点是用户数据分层与并发写控制
  - 需要区分基础数据、增量积分和大块文档数据
  - 需要用令牌或同类机制控制多端并发写
- 架构升级时，不要按“别人这么做”就直接上复杂模型：
  - 先看 PCU / CCU 和真实热点规模
  - 先看团队是否有维护复杂状态机的经验
  - 先看额外拆分带来的收益是否真能覆盖新增复杂度
