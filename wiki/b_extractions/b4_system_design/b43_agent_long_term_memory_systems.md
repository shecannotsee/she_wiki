# Agent 长期记忆系统设计

## 提取结果

- Agent 的长期记忆系统不应该只是聊天记录归档，而应该把对话沉淀为可检索、可复用、可服务当前动作的知识单元
- 一个可落地的记忆架构通常至少要区分工作记忆、语义记忆和程序性记忆，否则上下文管理、知识沉淀和技能复用会混在一起
- 记忆检索应该先围绕当前任务生成检索计划，再做多路召回和重排，而不是只用 embedding 相似度直接取 `top-k`
- 记忆固化应与回答链路异步解耦，并在写入前做新颖性判断与敏感信息过滤

## 核心结构

1. `Working Memory` 负责维护当前会话的活跃上下文，并在 token 预算逼近上限时压缩旧内容
2. `Semantic Memory` 负责保存稳定知识，例如事实、偏好、约束、事件、失败模式和工具能力
3. `Procedural Memory` 负责保存可复用的方法、流程和技能文档
4. 检索层先根据当前动作状态、缺失信息和时间约束生成计划，再做多通道召回与评分
5. 回答层使用融合后的背景上下文生成结果，长期记忆固化则在后台异步执行

## 设计要点

- 记忆条目应尽量做类型化和去上下文化，使其脱离原对话后仍可理解
- 排序目标除了语义相关性，还应纳入动作价值、槽位填补价值、时间匹配度、时效性和 token 成本
- 应优先返回融合后的高密度上下文，而不是把大量零散片段直接塞回模型
- 写入长期记忆前要先判断是否真有新信息，避免系统因重复固化而不断膨胀
- 把敏感信息过滤当成记忆系统的默认职责，而不是外部补丁

## 可直接复用

- 设计 Agent 记忆时，先问“哪些信息会影响下一步动作”，再决定记忆 schema
- 如果系统已经有对话压缩，总结层和长期记忆层不要混为一谈，它们服务的是不同时间尺度
- 检索接口最好拆成 `search -> synthesize -> reason` 三段，便于单独调试召回、融合和回答质量
- 若目标是工程落地，优先选择轻量存储与标准接入接口，而不是先引入高维护成本的复杂图数据库
