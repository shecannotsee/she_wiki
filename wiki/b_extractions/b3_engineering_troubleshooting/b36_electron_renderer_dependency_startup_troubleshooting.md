# Electron Renderer 依赖接入与启动链路排障顺序

## 提取结果

- 在 Electron 项目里，Renderer 能写成 TypeScript 和 ES Module，不代表它天然拥有 bundler 项目的依赖解析能力。
- 对 Renderer 入口链路新增第三方依赖时，最危险的错误之一，是把“编译通过”误当成“运行时一定可加载”。
- 一旦错误发生在入口模块、入口 HTML 或 preload 桥上，后果往往不是某个功能失效，而是整个前端界面无法初始化。
- 这类问题的稳定解法，不是先硬让某个库工作，而是先确认真实运行模型，再选择匹配的接入方式，并优先设计降级路径。

## 稳定排障顺序

1. 先定位改动发生在哪一层：`main`、`preload`、`renderer`、入口 HTML，还是普通业务模块。
2. 再确认当前 Renderer 的真实运行模型：
   - 是否经过 bundler
   - 是否是原生 ESM + `file://`
   - 是否开启 `nodeIntegration`
3. 再判断第三方依赖应该怎么接：
   - 直接 `import`
   - preload 暴露
   - 入口 HTML 全局脚本
4. 如果依赖已经进入启动链路，优先验证“页面能不能初始化”，而不是先盯类型检查和构建结果。
5. 对非硬依赖功能，优先改成存在时增强、缺失时降级，不让单一依赖把整个 Renderer 启动拖死。
6. 最后再回头优化接入方式、代码风格和功能完整性。

## 可直接复用

- 新依赖接入前，先问一句：当前这个运行位置到底像浏览器、像 Node，还是两者都不是。
- 在 `index.html`、Renderer 顶层模块、preload 桥附近改代码时，默认把它当成启动级风险处理。
- 不要把 `npm run check` 或 `npm run build` 通过，误当成 Electron 页面一定能正常启动。
- 如果浏览器环境无法解析裸模块名，就不要继续套 bundler 项目的接入习惯。
- 对启动链路上的增强功能，优先做：
  - 依赖存在时启用
  - 依赖缺失时局部降级
  - 整体页面仍可初始化
- 看到“窗口能打开但页面空白”时，优先怀疑入口模块加载失败，而不是先怀疑普通 UI 逻辑。
- Electron 排障里，模块路径、资源路径和运行边界的错误，通常比业务逻辑 bug 更容易把页面整体打死。
